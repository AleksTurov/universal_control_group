# Теория UKG Pipeline

## Что строит этот проект

Проект поддерживает `UKG` (`Universal Control Group`) для телекома.

Смысл UKG:

- есть единая постоянная контрольная группа на уровне всей eligible-базы;
- абонент после назначения не должен мигрировать между `control` и `test`;
- каждый месяц в assignment добавляются только новые абоненты, которых раньше не было в UKG-таблице;
- исторические назначения сохраняются и используются повторно.

Итоговая бизнес-идея такая:

- `control` это универсальный holdout;
- `test` это таргетируемая база;
- для каждой конкретной кампании потом берется релевантный срез из уже закрепленной глобальной группы.

## Основной принцип назначения

Единица назначения: `SUBS_ID`.

Назначение строится детерминированно:

1. для каждого нового абонента строится стабильный `split_hash` от пары `SUBS_ID | UKG_SALT`;
2. далее новые абоненты разбиваются на страты по набору бизнес-признаков;
3. внутри каждой страты часть клиентов уходит в `control`, остальные в `test`;
4. выбор внутри страты воспроизводим между запусками, пока не меняется `UKG_SALT`.

Это дает два свойства:

- один и тот же абонент всегда получает один и тот же hash при одинаковой соли;
- при повторном запуске на том же наборе новых клиентов assignment воспроизводим.

## Почему assignment persistent

В проекте используется не monthly reshuffle, а persistent assignment.

Это означает:

- если абонент уже был когда-то назначен в `control`, он и дальше считается `control`;
- если абонент уже был когда-то назначен в `test`, он и дальше считается `test`;
- в новый monthly insert попадают только строки без исторического назначения.

Такой подход нужен, чтобы:

- не ломать историю treatment/control;
- не мигрировать клиентов между группами;
- не создавать дополнительные смещения в аналитике;
- иметь стабильный baseline для CVM-коммуникаций.

## Как формируется monthly slice

Источник monthly slice задается SQL-файлом [sql/04_ukg_select_report_slice.sql](/data/aturov/universal_control_group/sql/04_ukg_select_report_slice.sql).

Логика такая:

1. `DATA_START` это первый день отчетного месяца;
2. `data_end` это последний день того же месяца;
3. eligibility берется из `DWH.hfct_subs_short` на конец месяца;
4. признаки для stratification и анализа берутся из `DWH.dm_datamart_monthly` на начало месяца;
5. уже существующее назначение подтягивается из `data_science.ukg_assignment`.

Из eligibility исключаются:

- `IZI`;
- сотрудники;
- тестовые абоненты;
- `M2M`.

На выходе SQL возвращает одну строку на `SUBS_ID` с:

- полями для stratification;
- numeric-полями для KS-проверок;
- историческими полями `existing_*`, если абонент уже был назначен раньше.

## Какие поля участвуют в стратификации

Текущий набор core-strata задается в [src/config.py](/data/aturov/universal_control_group/src/config.py):

- `ACTIVE_IND`
- `CUST_LEVEL`
- `REGION`
- `PERIODICITY`
- `FLAG_4G`
- `FLAG_ABONKA`
- `TENURE_BUCKET`
- `ARPU_BUCKET`
- `TRAFFIC_BUCKET`
- `MY_BEELINE_USER`

Эти поля сначала приводятся к строковому виду и склеиваются в единый `strata_key`.

Если в каком-то поле есть пропуск, он заменяется на `MISSING`, чтобы assignment был устойчивым и не зависел от формы `NULL/NaN`.

## Как строятся bucket-признаки

Bucket-логика реализована в [src/ukg_job.py](/data/aturov/universal_control_group/src/ukg_job.py).

Используются фиксированные бизнес-бакеты, а не monthly-квантили.

### TENURE_BUCKET

Строится по `LIFETIME_TOTAL`.

Границы:

- `0-30d`
- `31-90d`
- `91-180d`
- `181-365d`
- `1-2y`
- `2y+`

### ARPU_BUCKET

Строится по `REVENUE_TOTAL`.

Нули выделяются отдельно как `ARPU_ZERO`, а положительные значения режутся по фиксированным порогам.

### TRAFFIC_BUCKET

Строится по `USAGE_INTERNET`.

Нулевое использование идет в `TRAFFIC_ZERO`, положительные значения разбиваются на фиксированные интервалы.

## Как работает stratified split

Логика реализована в [src/stratified_assignment.py](/data/aturov/universal_control_group/src/stratified_assignment.py).

Алгоритм:

1. для новых клиентов строится `split_hash`;
2. строится `strata_key`;
3. для каждой страты считается ожидаемое число `control` как `n * UKG_PCT`;
4. сначала берется `floor` по каждой страте;
5. оставшийся хвост распределяется по стратам с максимальной дробной частью;
6. внутри каждой страты строки сортируются по `split_hash`, и первые `k` попадают в `control`.

Это схема largest remainder allocation.

Она нужна, чтобы:

- выдерживать глобальную целевую долю `UKG_PCT`;
- при этом сохранять баланс внутри страт;
- не использовать случайный `sample()` от запуска к запуску.

## Учебный пример: как считается split на простом игрушечном наборе

Ниже упрощенный пример, который можно читать как учебник.

Представим, что в новый monthly slice попали 10 новых абонентов. Мы хотим сделать `10%` control.

Допустим, после построения bucket-признаков получились такие strata:

- Страта A: `ACTIVE=1, REGION=BISHKEK, ARPU_BUCKET=ARPU_10_300, TRAFFIC_BUCKET=TRAFFIC_LIGHT`
- Страта B: `ACTIVE=1, REGION=OSH, ARPU_BUCKET=ARPU_300_500, TRAFFIC_BUCKET=TRAFFIC_HEAVY`

Пусть в страте A лежат 6 клиентов, а в страте B лежат 4 клиента.

Тогда целевое число control считается так:

$$
10 \times 0.10 = 1
$$

То есть глобально нам нужен 1 клиент в control.

Дальше внутри страт считаются ожидания:

$$
6 \times 0.10 = 0.6
$$

$$
4 \times 0.10 = 0.4
$$

Сначала берем `floor`:

- для страты A это 0
- для страты B это 0

Остался 1 нераспределенный control-клиент. Он идет в страту с большей дробной частью, то есть в страту A.

Дальше все клиенты внутри страты A сортируются по стабильному `split_hash`. Первый клиент по этому порядку становится `control`, остальные `test`.

Итог:

- в A будет `1 control` и `5 test`
- в B будет `0 control` и `4 test`

На маленьких данных это выглядит неровно, но на большой базе алгоритм дает нужную долю control и при этом сохраняет сопоставимость по ключевым baseline-осям.

## Учебный пример: почему persistent assignment важнее ежемесячного reshuffle

Представим абонента `SUBS_ID = 1001`.

В январе он впервые попал в eligible-базу и по hash-логике был назначен в `control`.

Что должно происходить дальше:

- в феврале, марте и следующих месяцах он не должен заново участвовать в split;
- он должен просто подтягиваться из historical assignment как уже закрепленный `control`;
- новый split должен считаться только для тех клиентов, которых раньше не было в assignment-таблице.

Если каждый месяц заново перемешивать всю базу, то один и тот же абонент мог бы:

- в январе быть `control`;
- в феврале стать `test`;
- в марте снова стать `control`.

Так делать нельзя, потому что это ломает экспериментальную историю и делает downstream-аналитику почти неинтерпретируемой.

## Почему были выбраны именно эти признаки

Выбор признаков опирается не на одну абстрактную идею, а на разбор в ноутбуках [notebook/02_EDA.ipynb](/data/aturov/universal_control_group/notebook/02_EDA.ipynb) и [notebook/03_UKG_split_KS.ipynb](/data/aturov/universal_control_group/notebook/03_UKG_split_KS.ipynb).

Логика выбора такая:

1. в mandatory-core попадают только `pre-treatment` признаки;
2. признаки должны быть интерпретируемыми для бизнеса;
3. признаки не должны взрывать число strata;
4. непрерывные поля лучше переводить в bucket-форму, а raw-распределения контролировать через `KS`.

Ниже практический смысл текущего набора.

### ACTIVE_IND

Почему нужен:

- активные и неактивные абоненты сильно различаются по вероятности отклика;
- это одна из самых сильных baseline-осей для CVM и churn-сценариев.

Почему в strata:

- бинарный и устойчивый признак;
- дает большой выигрыш в сопоставимости и почти не увеличивает fragmentation.

### CUST_LEVEL

Почему нужен:

- тип клиента влияет на продуктовый профиль, выручку и реакцию на коммуникации.

Почему в strata:

- категориальный и бизнес-осмысленный признак;
- важен почти для всех кампаний, а не только для одного узкого кейса.

### REGION

Почему нужен:

- регион влияет на покрытие, качество сети, продажи, каналы коммуникации и поведенческий профиль.

Почему в strata:

- это сильная baseline-ось;
- без нее группы могут разъехаться по географии еще до запуска кампаний.

### PERIODICITY

Почему нужен:

- периодичность тарифа отражает продуктовую механику и регулярность платежного поведения.

Почему в strata:

- дает интерпретируемое выравнивание по тарифной логике;
- умеренно кардинален.

### FLAG_4G

Почему нужен:

- это прокси технологической готовности абонента;
- влияет на доступность части digital и data-driven коммуникаций.

Почему в strata:

- бинарный и компактный признак;
- помогает не перекосить группы по устройственно-сетевой зрелости.

### FLAG_ABONKA

Почему нужен:

- отражает базовый платежный режим абонента;
- связан с monetization и типом продуктового поведения.

Почему в strata:

- простой и устойчивый baseline-фактор;
- хорошо интерпретируется в бизнесе.

### TENURE_BUCKET

Почему нужен:

- стаж клиента в сети почти всегда связан с churn-risk, платежной историей и устойчивостью поведения.

Почему именно bucket, а не raw `LIFETIME_TOTAL`:

- raw numeric быстро раздробит strata;
- bucket сохраняет смысл lifecycle, но остается устойчивым для split.

### ARPU_BUCKET

Почему нужен:

- monetization это одна из главных осей сравнимости `control/test`;
- сильный денежный baseline нельзя оставлять полностью без контроля.

Почему именно bucket, а не raw `REVENUE_TOTAL`:

- в ядро split лучше брать компактный monetization-профиль;
- само raw-распределение затем отдельно контролируется через `KS`.

### TRAFFIC_BUCKET

Почему нужен:

- интернет-активность разделяет клиентов по интенсивности использования и цифровому профилю.

Почему именно bucket:

- как и в случае с ARPU, raw traffic слишком детален для strata;
- bucket дает компактное, но полезное baseline-выравнивание.

### MY_BEELINE_USER

Почему нужен:

- это прокси готовности к app/digital-каналу;
- в ноутбуке он рассматривается как важная baseline-ось коммуникационного канала.

Почему в strata:

- бинарный и интерпретируемый признак;
- помогает не получить перекос по цифровой зрелости клиентов.

### Почему не все полезные признаки включены в ядро

В ноутбуке отдельно разбирается важный принцип: не каждое полезное поле должно попадать в mandatory-core.

Например, такие поля как:

- `TRANZ_FLAG`
- `BALANCE_USER`
- `MULTIPLAY`
- `FIRST_SIM`

могут быть бизнес-полезными, но каждое новое измерение увеличивает fragmentation.

Поэтому признаки делятся на роли:

- `mandatory core` для самого split;
- `KS-check` для post-split контроля;
- `candidate / enrichment` для отдельных экспериментов и пересмотра дизайна.

Это и есть причина, почему текущее ядро не максимальное, а компактное.

## Почему pre-insert checks считаются только на new_assignment_df

Это ключевой момент текущей логики.

Проверки до вставки считаются не на всем merged slice, а только на новых назначениях.

Причина:

- весь current slice содержит исторические назначения прошлых месяцев;
- в persistent UKG доля `control` в активной текущей базе со временем может естественно уходить от целевого `UKG_PCT`;
- если проверять SRM на всем merged slice, job будет ловить ложные ошибки при нормальной работе persistent assignment.

Поэтому pre-insert guard проверяет только то, что действительно создается в текущем запуске:

- новый monthly split;
- его SRM;
- его KS-баланс.

## Что проверяется до вставки

Pre-insert checks реализованы в [src/assignment_checks.py](/data/aturov/universal_control_group/src/assignment_checks.py).

До insert считаются:

1. `SRM` по `new_assignment_df`
2. `KS` по выбранным numeric-признакам тоже по `new_assignment_df`

Если одна из проверок не проходит, job падает до записи в ClickHouse.

Это означает, что в production-таблицу не должен попадать monthly split с явным перекосом по группам.

## Как контролируются дубли

Целевое правило такое:

- в `dm_datamart_monthly` должен быть один `SUBS_ID` на дату `DT`;
- в `ukg_assignment` для нового monthly insert не должно быть повторных `SUBS_ID`.

На практике защита сейчас двухслойная:

1. в SQL slice возвращается одна строка на `SUBS_ID`;
2. перед insert в [src/database.py](/data/aturov/universal_control_group/src/database.py) есть hard-check на дубли в `new_assignment_df`.

Если в новых назначениях найден повторный `SUBS_ID`, запись отменяется исключением.

Важно понимать:

- `ReplicatedMergeTree` и `Distributed` в ClickHouse не дают настоящего `UNIQUE CONSTRAINT`;
- поэтому логическая уникальность обеспечивается пайплайном, а не DDL движка.

## Как происходит insert

DDL задается в [sql/03_ukg_assignment_monthly.sql](/data/aturov/universal_control_group/sql/03_ukg_assignment_monthly.sql).

Используются две таблицы:

- `data_science.ukg_assignment_src` на `ReplicatedMergeTree`
- `data_science.ukg_assignment` на `Distributed`

В Python insert идет в ORM-модель `UkgAssignment`, которая привязана к distributed-таблице.

Зачем это сделано:

- вставка выполняется в одну логическую таблицу;
- ClickHouse сам разносит строки по шардам;
- аналитический доступ идет через одну distributed-таблицу.

## Что хранится в assignment-таблице

В таблице сохраняются:

- `subs_id`
- `first_seen_dt`
- `assignment_dt`
- `experiment_group`
- `is_control`
- `split_hash`
- `ukg_pct`
- `ukg_salt`
- `assignment_version`
- `created_at`

Также на стороне DDL есть materialized-поля с бизнес-названиями группы.

Это позволяет использовать assignment не только как техническую таблицу split, но и как источник для downstream-процессов исключения из коммуникаций.

## Что делает app.py

Основной orchestration лежит в [src/app.py](/data/aturov/universal_control_group/src/app.py).

Пайплайн выглядит так:

1. прочитать monthly slice из SQL;
2. достроить bucket-признаки;
3. выделить только новых клиентов без `existing_experiment_group`;
4. выполнить stratified assignment;
5. прогнать pre-insert checks на новых назначениях;
6. построить ORM-модели;
7. вставить новые строки в ClickHouse, если `DRY_RUN = False`;
8. собрать merged current slice из старых и новых назначений;
9. посчитать post-run анализ и сохранить артефакты.

## Что считается после вставки

После insert pipeline строит merged current slice:

- старые historical назначения сохраняются;
- новые назначения добавляются поверх текущего monthly slice.

На этом merged slice выполняются:

- run summary;
- графики;
- SQL validation queries;
- диагностические артефакты для аудита запуска.

Это уже не блокирующий слой, а слой наблюдаемости и контроля качества результата.

## Почему status run не должен зависеть от дублей во всей historical assignment-таблице

Раньше была идея поднимать warning, если в global assignment summary есть дубли по всей таблице.

Сейчас это убрано из run status по двум причинам:

1. status текущего запуска должен отражать качество именно текущего monthly split;
2. глобические historical проблемы нельзя смешивать с результатом конкретного запуска.

Поэтому итоговый `status` run сейчас зависит только от:

- `SRM`;
- `KS`.

## Практические правила эксплуатации

1. Не менять `UKG_SALT`, если не планируется полная миграция assignment-логики.
2. Любое изменение логики split фиксировать через `ASSIGNMENT_VERSION`.
3. Не делать reshuffle уже назначенных абонентов.
4. Перед production insert сначала проверять monthly split через pre-insert guard.
5. Передавать `report_dt` в формате `YYYY-MM-DD`.

## Что делать при перезапуске того же месяца

Для repeated run одного и того же `report_dt` правильная эксплуатационная политика такая:

1. сначала удалить из assignment все строки с этим `assignment_dt`;
2. затем заново вставить пересчитанный monthly batch;
3. historical месяцы при этом не трогать.

Почему так:

- persistent assignment должен сохранять историю прошлых месяцев;
- но конкретный monthly batch должен быть переигрываемым, если запуск пришлось пересчитать;
- delete + reinsert для одного месяца делает rerun понятным и контролируемым.

То есть логика не такая:

- "пересчитать всю таблицу с нуля";

а такая:

- "переиграть только проблемный monthly слой".

Практически это означает, что при rerun за `2026-02-01` мы должны удалить только строки с `assignment_dt = '2026-02-01'`, а затем заново вставить новый batch за этот месяц.

## Что делать, если в будущем control и test начнут различаться сильнее

Это нормальный вопрос для persistent holdout.

Со временем active база почти неизбежно начнет расходиться между `control` и `test`, потому что:

- клиенты churn-ятся несимметрично;
- eligibility меняется каждый месяц;
- разные типы клиентов по-разному выпадают из базы;
- часть исторических сегментов может становиться редкой.

Главное правило здесь такое:

- не переписывать исторический assignment полностью только потому, что current active base стала менее симметричной.

Полный rewrite всей таблицы это крайняя мера, потому что он:

- ломает историческую экспериментальную историю;
- делает прошлые кампании несопоставимыми;
- фактически создает новую UKG-систему, а не корректирует старую.

Правильная стратегия обычно идет по уровням.

### Уровень 1. Наблюдать drift

Сначала нужно понимать, что именно разъехалось:

- только active current slice;
- только отдельные сегменты;
- или уже сам monthly split перестает быть сбалансированным на новых клиентах.

Если monthly split на новых клиентах все еще хороший, а разъезд виден только на исторической active базе, это не означает, что систему надо переписывать.

### Уровень 2. Не трогать historical assignment, но усилить monitoring

Если drift умеренный, стандартное решение такое:

- historical assignment не переписываем;
- продолжаем делать качественный split только для новых клиентов;
- отдельно мониторим drift по важным бизнес-срезам.

Это нормальная стратегия для persistent holdout.

### Уровень 3. Пересмотреть ядро strata для новых месяцев

Если видно, что какая-то важная baseline-ось системно недоконтролируется, можно:

- изменить ядро strata;
- изменить bucket-дизайн;
- зафиксировать новую логику через новый `ASSIGNMENT_VERSION`.

Важно:

- это должно применяться к новым monthly назначениям;
- старую историю обычно не переписывают задним числом.

### Уровень 4. Создать новую версию UKG только если старый дизайн уже методологически непригоден

Полный restart или полная пересборка нужны только в тяжелом случае, когда:

- сильно изменился бизнес-контекст;
- изменилась сама eligible-база;
- старое ядро strata перестало отражать важные baseline-оси;
- historical design больше нельзя интерпретировать как корректный holdout.

Тогда обычно делают не "тихий rewrite", а явную новую версию системы:

- новая соль;
- новая версия assignment;
- новая документация;
- отдельная дата старта новой UKG.

Иначе говоря, правильный ответ обычно не "переписывать полностью при первом drift", а:

1. мониторить;
2. при необходимости адаптировать только новый monthly split;
3. historical assignment не переписывать без очень сильной причины.

## FAQ

### Вопрос: зачем вообще нужен persistent UKG, а не новый random split каждый месяц?

Короткий ответ: чтобы один и тот же абонент не мигрировал между `control` и `test`.

Если каждый месяц перемешивать всю базу заново, ломается история воздействия, и прошлые кампании перестают быть сопоставимыми.

### Вопрос: почему pre-insert checks считаются только на новых клиентах?

Потому что именно новые клиенты назначаются в текущем запуске.

Историческая active база со временем может естественно разъехаться из-за churn, смены eligibility и структуры клиентов. Это не означает, что текущий monthly split плохой.

### Вопрос: что считается после вставки?

После insert считается monitoring на полном текущем assignment slice месяца:

- с учетом старых historical назначений;
- с учетом новых вставленных строк;
- с расчетом SRM, KS, summary и SQL validations.

Именно этот слой нужен, чтобы видеть, как выглядит вся текущая база после объединения старых и новых assignment-строк.

### Вопрос: если на всей базе control и test со временем разъедутся, нужно ли переписывать всю таблицу?

Обычно нет.

Правильный порядок действий такой:

1. проверить, хорош ли monthly split на новых клиентах;
2. понять, разъехалась ли только active historical база или уже ломается новый split;
3. если ломается только historical current slice, усилить monitoring и при необходимости обновить strata-дизайн только для новых месяцев;
4. полную пересборку UKG делать только если текущая схема уже методологически непригодна.

Иными словами, drift на всей базе сам по себе не означает, что нужно переписать всю историю.

### Вопрос: как правильно перезапускать месяц?

Правильная схема такая:

1. удалить строки из assignment с этим `assignment_dt`;
2. заново посчитать batch за этот же месяц;
3. вставить новый batch;
4. historical месяцы не трогать.

Именно поэтому rerun месяца должен работать как `delete + reinsert`, а не как полная пересборка всей таблицы.

## Ограничения текущей схемы

1. Логическая уникальность `SUBS_ID` не обеспечивается native constraint со стороны ClickHouse.
2. Корректность monthly slice зависит от того, что `dm_datamart_monthly` действительно содержит одну строку на `SUBS_ID` и `DT`.
3. Баланс проверяется только по новым назначениям, а не по всей активной исторической базе.

Это не баг, а осознанный дизайн для persistent assignment.

## Короткая формула всей логики

Если свести весь pipeline к одной формуле, то он работает так:

$$
\text{Current UKG} = \text{Historical Assignment} \cup \text{New Monthly Assignment}
$$

где:

$$
\text{New Monthly Assignment} = \text{Deterministic Stratified Split on New Eligible Subscribers}
$$

А значит, система не пересчитывает прошлое, а только аккуратно доращивает уже существующую глобальную контрольную группу.
